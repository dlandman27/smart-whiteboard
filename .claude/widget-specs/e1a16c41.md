# Sports Widgets — Full Visual Redesign

**One-liner:** Redesign all sports widgets (NFL, NBA, and any future leagues) to be fully responsive, image-visible, properly grouped, and scroll-free within their default size.

---

## Background / Problem

The current `SportsWidget.tsx` has several compounding issues:
1. **Team logos don't render** — `img` tags use ESPN logo URLs that are blocked or fail silently; no fallback shown.
2. **Horizontal scrollbar appears** — `FlexRow` inside `ScrollArea` overflows horizontally because `minWidth` values on team columns fight the container width.
3. **No grouping** — Live, upcoming, and final games are sorted but not visually separated into sections.
4. **Doesn't scale** — Font sizes, logo sizes, and padding are hardcoded px values; the widget looks broken at non-default sizes.
5. **General aesthetic** — The layout feels cramped and unpolished compared to other widgets (e.g. WeatherWidget uses `ResizeObserver` for fluid scaling).

Affected widgets: `@whiteboard/nfl`, `@whiteboard/nba` (and any future leagues added to `ESPN_URLS` in `server/routes/sports.ts`).

---

## User Stories

1. **As a user**, I can see team logos (or a clean fallback abbreviation badge) so I can identify teams at a glance without reading text.
2. **As a user**, I can see games grouped by status (🔴 Live · Upcoming · Final) so I immediately know what's happening now vs. later.
3. **As a user**, I can resize the sports widget freely and the content scales proportionally — no horizontal scrollbar, no clipped text.
4. **As a user**, I can pick a favorite team in settings and that team's game is visually pinned/highlighted at the top of its group.

---

## Acceptance Criteria

### Layout & Overflow
- [ ] No horizontal scrollbar at any widget width ≥ 240 px.
- [ ] No vertical scrollbar visible at the default size (340 × 420). Vertical scroll is allowed only when the game count genuinely exceeds the container height.
- [ ] Widget uses `ResizeObserver` (same pattern as `WeatherWidget`) to derive fluid font sizes, logo sizes, and padding from container width.
- [ ] All text truncates with `text-overflow: ellipsis` rather than wrapping or overflowing.

### Team Logos / Images
- [ ] Logo `<img>` elements have `crossOrigin="anonymous"` and an `onError` handler.
- [ ] On image load failure, render a circular badge with the team abbreviation (2–3 chars) using `var(--wt-surface-raised)` background and `var(--wt-text-muted)` text — **no broken image icon**.
- [ ] Logo size scales with container width: `clamp(18px, 6%, 32px)`.

### Grouping
- [ ] Games are rendered in three labeled sections: **Live** (status `in`), **Upcoming** (status `pre`), **Final** (status `post`).
- [ ] Sections with zero games are hidden entirely (no empty headers).
- [ ] Section headers use `var(--wt-text-muted)` color, uppercase, small caps style — consistent with other widget headers.
- [ ] Live section header includes a pulsing red dot (reuse existing pulse animation).

### Scaling
- [ ] At width < 280 px: hide win/loss records; show only abbr + score.
- [ ] At width ≥ 280 px: show records for pre-game rows.
- [ ] Font sizes derived from container width using `clamp()` or `ResizeObserver`-driven state (match WeatherWidget pattern).

### Colors & Theming
- [ ] **Zero hardcoded color values** — all colors via CSS vars (`var(--wt-accent)`, `var(--wt-text)`, `var(--wt-text-muted)`, `var(--wt-border)`, `var(--wt-surface)`, `var(--wt-surface-raised)`).
- [ ] Exception: the live-indicator red dot may use `var(--wt-danger)` or the existing `#ef4444` only if a CSS var is unavailable — prefer `var(--wt-danger)`.
- [ ] Favorite team highlight uses `color-mix(in srgb, var(--wt-accent) 10%, transparent)` background (existing pattern is fine, just ensure no hardcoded colors elsewhere).

### Icons
- [ ] All non-logo icons (e.g. live dot, section chevrons if added) use Phosphor icons via the existing `<Icon>` component.

### Registry
- [ ] Default sizes remain `{ width: 340, height: 420 }` for both NFL and NBA — the redesign must look good at this size without scroll.
- [ ] Both `NFLWidget` and `NBAWidget` remain registered in `registry.tsx` with no type/label changes.
- [ ] `SportsSettings` (favorite team picker) continues to work unchanged.

### Data / Hook
- [ ] No changes to `useSports.ts` hook or `server/routes/sports.ts` — this is a pure UI redesign.
- [ ] The shared `SportsShell` component is refactored to accept a `league` prop and render the new layout; both `NFLWidget` and `NBAWidget` use it.

### Accessibility
- [ ] Each game row has `role="listitem"` and the game list has `role="list"`.
- [ ] Logo `<img>` elements have `alt="{team.abbr} logo"`; fallback badges have `aria-label="{team.abbr}"`.
- [ ] Live status text is wrapped in `<span aria-live="polite">` so screen readers announce score changes.

## Default Size
- **Width:** 340 px  
- **Height:** 420 px  
*(unchanged from current registry entries)*

## Settings / Preferences
- **Favorite Team** (existing) — team abbreviation string; game is pinned to top of its group and highlighted.
- No new settings required for this redesign.

## Data Sources
- ESPN Scoreboard API via `/api/sports/:league` (proxied by `server/routes/sports.ts`)
- Polled every 60 s via `useNFLScores` / `useNBAScores` hooks (unchanged)

---

## Out of Scope
- Adding new leagues (MLB, NHL, soccer) — server routes exist but widget registration is a separate task.
- Click-through to game detail pages.
- Push notifications for score changes.
- Dark/light mode toggle (handled globally by CSS vars).
- Any changes to `SportsSettings.tsx` beyond what's needed to keep it working.
- Performance optimization of the ESPN API proxy.

## Open Questions
1. **Logo CORS** — ESPN logo URLs (`a.espncdn.com`) may block cross-origin requests. Should the server proxy logo images, or is the `onError` fallback badge sufficient? *(Defer to Lead Engineer — this is an architecture/reliability call.)*
2. **Shared component** — Should `SportsShell` live in `SportsWidget.tsx` or be extracted to a `sports/` subdirectory in anticipation of MLB/NHL widgets? *(Defer to Lead Engineer.)*
3. **Vertical scroll at small heights** — If a user resizes the widget to a very short height, should we show a "X more games" collapsed state, or just let it scroll? *(Product decision — default: allow scroll, no collapse for now.)*
4. **`var(--wt-danger)` availability** — Confirm this CSS var exists in the theme before using it for the live dot; fallback to `#ef4444` if not.
